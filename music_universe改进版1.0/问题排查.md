# Music Universe（音乐宇宙）改进版 — 问题排查

本文按「现象 → 可能原因 → 处理步骤」组织，便于在课程答辩或本地复现时快速定位问题。

---

## 1. Python 分析脚本：`music_hit_analysis.py`

### 1.1 报错：找不到 CSV 或路径在另一台电脑上无效

**现象**：`FileNotFoundError` 或日志里出现 `D:\File\R\...` 等本机不存在的路径。

**原因**：脚本内默认路径写死为示例 Windows 目录。

**处理**：

1. 设置环境变量指向本仓库数据（路径按你的实际位置修改）：
   - `MUSIC_SPOTIFY_PATH`
   - `MUSIC_BILLBOARD_PATH`
   - `MUSIC_TOPICS_PATH`
2. 如需自定义输出位置，设置 `MUSIC_OUTPUT_ROOT`。

在 macOS / Linux 下示例：

```bash
# 路径请按本机实际位置修改（以下为仓库名 music_universe改进版1.0 示例）
export MUSIC_SPOTIFY_PATH="/绝对路径/music_universe改进版1.0/data/spotify_tracks.csv"
export MUSIC_BILLBOARD_PATH="/绝对路径/music_universe改进版1.0/data/billboard.csv"
export MUSIC_TOPICS_PATH="/绝对路径/music_universe改进版1.0/data/topics.csv"
export MUSIC_OUTPUT_ROOT="/绝对路径/music_universe改进版1.0/output"
python code/music_hit_analysis.py
```

### 1.2 报错：无法读取 CSV / 乱码

**现象**：`RuntimeError: 无法读取文件` 或列名、中文显示为乱码。

**原因**：源文件编码与 `pandas` 默认不一致。

**处理**：脚本已按 `utf-8-sig` → `utf-8` → `gbk` → `latin1` 顺序尝试；若仍失败，用编辑器将 CSV 另存为 **UTF-8（带 BOM 或无 BOM 均可）** 后重试。查看 `output/logs/music_hit_analysis_v3.log` 中「读取成功」一行确认实际使用的 encoding。

### 1.3 某张图保存失败或中途中断

**现象**：日志停在某一 `run_plot` 步骤，或内存占用陡增。

**原因**：数据量极大（如 Spotify 全表）时，个别密度图/矩阵运算耗内存；磁盘权限或路径不可写。

**处理**：

- 确认 `MUSIC_OUTPUT_ROOT` 可写且磁盘空间充足。
- 缩小输入数据行数做冒烟测试。
- 根据日志中的异常栈定位具体绘图函数，必要时降低分辨率或采样（需改代码时保持改动范围最小）。

### 1.4 摘要报告里 Spotify 与 Billboard 数值「不在同一量级」

**现象**：例如 `analysis_summary_report.md` 中 danceability 一行类似：Spotify ≈ 0.59，Billboard ≈ 63。

**原因**：Spotify 音频特征在数据集中多为 **0–1** 连续值；Billboard 数据集中 `danceability`、`energy` 等字段来自另一套量表，常为 **0–100** 量级。对照表若未统一归一化，会出现数值不可比。

**处理**：

- 写论文或展示时 **不要直接并排解读绝对数值**，应说明量纲差异，或对 Billboard 列除以 100 后再对比。
- 若需严谨对比，应在 `build_comparison_table` 一类逻辑中增加显式缩放并注明。

## 2. 元数据增强脚本：`enrich_music_metadata.py`

### 2.1 报错：`does not start with 'window.SITE_DATA = '`

**现象**：`ValueError: ... does not start with ...`

**原因**：`music_universe_data_embedded.js` 被手工编辑或导出格式改变，丢失了脚本要求的单行前缀。

**处理**：确保文件开头严格为：

```text
window.SITE_DATA = {...JSON...};
```

JSON 部分需可被 `json.loads` 解析；末尾分号可有可无（脚本会处理）。

### 2.2 请求很慢、频繁失败或被封禁

**现象**：终端长时间无进度；`cache` 里 `found: false` 增多；`HTTPError` / timeout。

**原因**：iTunes / 公共 API 有 **速率限制**；并发过高或 `sleep` 过小易触发限流；网络不稳定。

**处理**：

- 增大 `--sleep`（例如 0.15–0.3），减小 `--workers`。
- 使用 `--limit` 分批补全，避免一次跑完全库。
- 查看 `output/tables/itunes_metadata_cache.json` 中条目的 `error` 字段。
- 确认本机可访问外网；若在校网环境，检查代理或防火墙。

### 2.3 匹配到错误歌曲或封面/预览不对

**原因**：标题、艺人重名；查询词被特殊字符干扰；不同地区店面结果不同。

**处理**：

- 尝试 `--country`（如 `US`、`JP`）看匹配是否改善。
- 使用 `--retry-failed` 对无预览缓存重查。
- 对关键展示曲目可在数据源中修正 `title`/`artist` 拼写后再跑增强脚本。

## 3. Web 前端：`index.html` + `app.js`

### 3.1 页面空白或提示「Failed to load Three.js or OrbitControls」

**现象**：`errorBanner` 显示加载失败；控制台报脚本 404。

**原因**：

- 未使用 HTTP 服务，相对路径 `./vendor/three.min.js` 解析异常。
- `vendor` 目录缺失且 CDN 也被拦截（离线、企业代理）。

**处理**：

1. 在项目根或 `web/music_universe/` 下启动静态服务，任选其一：
   ```bash
   python web/music_universe/serve_web.py
   ```
   或：
   ```bash
   cd web/music_universe && python -m http.server 8080
   ```
   浏览器访问 `http://127.0.0.1:8080/`（若改了 `PORT` 环境变量则换端口）。
2. 确认 `vendor/three.min.js` 与 `vendor/OrbitControls.js` 存在。
3. 允许浏览器访问 CDN（`cdnjs`、`jsdelivr`），或改为只使用本地 vendor。

### 3.2 提示「脚本加载失败」且提到 `music_universe_data_embedded`

**现象**：`index.html` 内联 bootstrap 捕获到 `load failed`。

**原因**：`music_universe_data_embedded.js` 缺失、路径错误，或文件过大导致解析超时（较少见）、或 JS 语法错误（内嵌 JSON 未转义）。

**处理**：

- 确认 `web/music_universe/music_universe_data_embedded.js` 与 `index.html` 同目录。
- 若改用 JSON 管线，放置 `music_universe_data_embedded.json` 并确保 **不** 先加载错误的内嵌脚本（需与当前 `index.html` 加载顺序一致）。
- 用 JSON 校验工具检查导出内容是否合法。

### 3.3 `loadSiteData` 报 `Failed to load data: 404`

**现象**：控制台显示对 `music_universe_data_embedded.json` 的 fetch 失败。

**原因**：仅加载了 `app.js` 而未先注入 `window.SITE_DATA`，且仓库中无对应 JSON 文件。

**处理**：按默认设计先加载 `music_universe_data_embedded.js`；或自行从分析结果生成 JSON 并命名为上述文件名。

### 3.4 点击「Play Preview」无声音或立即报错

**现象**：控制台 CORS 相关错误；或浏览器提示需用户手势激活音频。

**原因**：

- 试听 URL 所在域未对 `crossOrigin` 开放，或与 `HTMLAudioElement` 策略冲突。
- 未先点击页面即自动播放，被策略拦截。

**处理**：

- 优先在用户点击「Play Preview / Play Signature」后再播放（当前逻辑已偏向点击触发）；可尝试刷新后 **先点击页面任意处** 再点播放。
- 若无公开 `preview_url`，使用「Play Signature」走 Web Audio 路径。
- 更换浏览器测试（Chrome / Edge 对 Web Audio 支持较完整）。

### 3.5 专辑封面不显示或裂图

**原因**：`image_url` / `artwork_url` 缺失；外链拒绝 hotlink；CORS。

**处理**：运行 `enrich_music_metadata.py` 补全元数据；检查 `artBox` 网络请求是否 403；必要时将封面缓存到本地静态目录并改数据中的 URL（需额外开发）。

## 4. 数据与结论展示

### 4.1 Matplotlib 日志：`Using categorical units to plot a list of strings...`

**现象**：日志中出现该 INFO，不一定失败。

**原因**：部分刻度被推断为分类变量。

**处理**：若图已正常生成可忽略；若轴标签错误，在对应绘图函数中将列显式转为数值类型。

### 4.2 交互 Plotly HTML 打不开或中文乱码

**处理**：用现代浏览器直接打开 `output/figures/interactive_html/*.html`；系统缺少中文字体时，将图表中文字体改为网页安全字体或在生成代码中指定字体路径（属脚本增强范畴）。

---

## 5. 建议的排错顺序（清单）

1. 读 `output/logs/music_hit_analysis_v3.log` 最后数十行。  
2. 浏览器 **开发者工具 → Console / Network**。  
3. 确认 **环境变量路径** 与 **静态服务** 工作目录。  
4. 核对 **`window.SITE_DATA = `** 前缀与 JSON 合法性。  
5. 区分 **0–1 与 0–100 量纲** 后再写结论。

如以上仍无法解决，把 **完整报错栈**、**操作系统与 Python 版本**、以及 **是否使用代理** 一并记录，便于他人复现。
